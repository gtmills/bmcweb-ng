auth,tls: implement mutual TLS (mTLS) client certificate authentication

Add mTLS support across config, TLS layer, and auth middleware.

config/mod.rs:
  - New ServerConfig fields: mtls_enabled (bool, default false) and
    mtls_ca_cert (string, default /etc/bmcweb/ca.pem)
  - #[serde(default)] on both fields so existing config.toml files
    without these fields continue to work unchanged

protocol/http.rs:
  - build_mtls_config(cert, key, ca) — builds a rustls ServerConfig
    with WebPkiClientVerifier requiring client certs signed by the CA
  - run_tls() selects build_mtls_config or build_tls_config based on
    mtls_enabled; logs whether mTLS is active at startup
  - extract_peer_cert_subject() — after handshake, reads
    peer_certificates() from the rustls ServerConnection and extracts
    the Subject CommonName via DER byte walk (OID 2.5.4.3 / 0x55 0x04 0x03)
  - Peer cert CN injected as X-Client-Cert-Subject request header
    before dispatching to axum handlers
  - extract_cn_from_der() helper exposed for unit testing

auth/middleware.rs:
  - New auth arm: checks X-Client-Cert-Subject header (set by the TLS
    accept loop above); if present, creates a session using the CN as
    the username — cert is already verified by rustls CA trust store,
    no password needed

3 new unit tests in http.rs:
  test_extract_cn_from_der_empty, test_extract_cn_from_der_simple,
  test_build_mtls_config_missing_ca

128 unit tests passing; zero clippy warnings.
